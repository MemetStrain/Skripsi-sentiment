Pendahuluan

Proses P5 — Dashboard & Live Inference dari DFD Level 1 dimulai dari perintah python manage.py runserver yang dijalankan oleh Operator untuk menyalakan Django development server pada 127.0.0.1:8000. Secara internal, perintah CLI ini memicu rantai boot sequence Django: manage.main() menetapkan DJANGO_SETTINGS_MODULE=config.settings, kemudian mendelegasikan ke execute_from_command_line() yang me-resolve command runserver ke django.core.management.commands.runserver.Command. Command tersebut membangun WSGI handler via get_wsgi_application() (website/config/wsgi.py) dan mem-bind socket pada port 8000. Setelah server siap, setiap HTTP request dari Pengunjung dikirim ke URL dispatcher (config/urls.py → web/urls.py) yang me-route ke salah satu dari empat fungsi view publik berdasarkan path.

Karena rantai boot sequence dan URL dispatching adalah infrastruktur Django framework — bukan modul aplikasi yang ditulis pengembang — structure chart ini menyederhanakan keseluruhan rantai tersebut menjadi satu akar tunggal python manage.py runserver. Empat view publik (dashboard, news, about, forecasts_api) digambarkan sebagai modul anak langsung dari akar, dengan label panah menyatakan HTTP method + path yang men-trigger pemanggilan. Dekomposisi lebih lanjut di bawah setiap view menampilkan modul-modul aplikasi yang sebenarnya: loader Firestore, feature engineering, dan inferensi XGBoost dari saved models. Detail modul boot Django tetap didokumentasikan pada tabel “Modul Boot & Dispatch” di bawah untuk kelengkapan referensi.

Catatan Implementasi

  • Akar = CLI tunggal: sesuai arahan pembimbing, akar structure chart adalah python manage.py runserver sebagai satu kotak. Rantai Django boot sequence (manage.main → execute_from_command_line → Command → WSGIHandler → URL dispatcher) dijelaskan pada paragraf Pendahuluan dan tabel “Modul Boot & Dispatch”, tetapi tidak digambar di diagram karena merupakan infrastruktur framework, bukan modul aplikasi.
  • Tanpa autentikasi: seluruh endpoint publik read-only.
  • Inferensi live: forecasts_api memuat model.pkl + scaler.pkl per (tag, horizon) via joblib.load, di-cache process-wide dengan lru_cache(64).
  • Single source of truth fitur: feature_engineering.py di prediction/ dipakai bersama oleh skrip training offline (horizon_forecast_C{1..4}_*.py) dan path inferensi website agar tidak terjadi divergensi train/serve.
  • Composite-index avoidance: news meng-fetch semua doc dari news_articles lalu sort di Python, untuk menghindari kebutuhan composite index Firestore (date + sentiment).
  • Data store yang dibaca P5: D1 (daily_prices), D3 (sentiment_aggregates), D4 (hmm_states), D2 (news_articles), D6 (saved_models/*.pkl), D7 (winners.json).
