Pendahuluan

Proses 6.0 — Dashboard Web dari DFD Level 1 dimulai dari perintah python manage.py runserver yang dijalankan oleh Operator untuk menyalakan Django development server pada 127.0.0.1:8000. Secara internal, perintah CLI ini memicu rantai boot sequence Django: manage.main() menetapkan DJANGO_SETTINGS_MODULE=config.settings, kemudian mendelegasikan ke execute_from_command_line() yang me-resolve command runserver ke django.core.management.commands.runserver.Command. Command tersebut membangun WSGI handler via get_wsgi_application() (website/config/wsgi.py) dan mem-bind socket pada port 8000. Setelah server siap, setiap HTTP request dari Pengunjung dikirim ke URL dispatcher (config/urls.py → web/urls.py) yang me-route ke salah satu dari empat fungsi view publik berdasarkan path.

Karena rantai boot sequence dan URL dispatching adalah infrastruktur Django framework — bukan modul aplikasi yang ditulis pengembang — structure chart ini menyederhanakan keseluruhan rantai tersebut menjadi satu akar tunggal python manage.py runserver. Empat view publik (dashboard, news, about, forecasts_api) digambarkan sebagai modul anak langsung dari akar, dengan label panah menyatakan HTTP method + path yang men-trigger pemanggilan. Dekomposisi lebih lanjut di bawah setiap view menampilkan modul-modul aplikasi yang sebenarnya: loader Firestore dan render template. Keempat view bersifat read-only — tidak ada inferensi XGBoost di jalur request. dashboard membaca forecast_meta/Daily (D9) untuk badge metrik h=1 dan tabel komparasi 4×7; forecasts_api membaca forecast_meta/Daily ditambah seluruh koleksi forecasts (D8, satu dokumen per anchor × horizon) dan merakit ulang payload trail yang sama dengan jalur lama. Kedua store diproduksi oleh proses 5.0 (Forecast Precompute) di scheduler — modul scheduler/precompute_forecasts.py + algoritma di prediction/inference.py. Detail modul boot Django tetap didokumentasikan pada tabel “Modul Boot & Dispatch” di bawah untuk kelengkapan referensi.

Catatan Implementasi

  • Akar = CLI tunggal: sesuai arahan pembimbing, akar structure chart adalah python manage.py runserver sebagai satu kotak. Rantai Django boot sequence (manage.main → execute_from_command_line → Command → WSGIHandler → URL dispatcher) dijelaskan pada paragraf Pendahuluan dan tabel “Modul Boot & Dispatch”, tetapi tidak digambar di diagram karena merupakan infrastruktur framework, bukan modul aplikasi.
  • Tanpa autentikasi: seluruh endpoint publik read-only.
  • forecasts_api read-only: endpoint /api/forecasts/ hanya membaca forecast_meta/Daily + koleksi forecasts dari Firestore — tidak ada joblib.load, model XGBoost, atau scaler di jalur request. Filter predicted_date dan horizon dilakukan di Python untuk menghindari kebutuhan composite index Firestore.
  • Inferensi dipre-komputasi oleh scheduler: trail forecast diproduksi oleh proses 5.0 (scheduler/precompute_forecasts.py) yang memanggil prediction/inference.py::compute_forecast_trails dan menulis hasilnya ke D8 (forecasts) + D9 (forecast_meta/Daily). prediction/feature_engineering.py tetap menjadi single source of truth fitur, dipakai bersama skrip training horizon_forecast_C{1..4}_*.py dan inference — agar tidak terjadi divergensi train/serve.
  • Composite-index avoidance: news meng-fetch semua doc dari news_articles lalu sort di Python; forecasts_api juga men-stream seluruh koleksi forecasts lalu filter di Python. Pola yang sama menghindari kebutuhan composite index Firestore.
  • Data store yang dibaca P6: D1 (daily_prices), D2 (news_articles), D3 (sentiment_aggregates), D4 (hmm_states), D8 (forecasts per-point), D9 (forecast_meta/Daily).
