Pendahuluan

Proses P5 — Dashboard Web dari DFD Level 1 dimulai dari perintah python manage.py runserver yang dijalankan oleh Operator untuk menyalakan Django development server pada 127.0.0.1:8000. Secara internal, perintah CLI ini memicu rantai boot sequence Django: manage.main() menetapkan DJANGO_SETTINGS_MODULE=config.settings, kemudian mendelegasikan ke execute_from_command_line() yang me-resolve command runserver ke django.core.management.commands.runserver.Command. Command tersebut membangun WSGI handler via get_wsgi_application() (website/config/wsgi.py) dan mem-bind socket pada port 8000. Setelah server siap, setiap HTTP request dari Pengunjung dikirim ke URL dispatcher (config/urls.py → web/urls.py) yang me-route ke salah satu dari empat fungsi view publik berdasarkan path.

Karena rantai boot sequence dan URL dispatching adalah infrastruktur Django framework — bukan modul aplikasi yang ditulis pengembang — structure chart ini menyederhanakan keseluruhan rantai tersebut menjadi satu akar tunggal python manage.py runserver. Empat view publik (dashboard, news, about, forecasts_api) digambarkan sebagai modul anak langsung dari akar, dengan label panah menyatakan HTTP method + path yang men-trigger pemanggilan. Dekomposisi lebih lanjut di bawah setiap view menampilkan modul-modul aplikasi yang sebenarnya: loader Firestore dan render template. Keempat view bersifat read-only — tidak ada inferensi XGBoost di jalur request; forecasts_api hanya membaca dokumen Firestore forecasts/latest (D8) yang diproduksi oleh skrip offline website/precompute_forecasts.py. Detail modul boot Django tetap didokumentasikan pada tabel “Modul Boot & Dispatch” di bawah untuk kelengkapan referensi.

Catatan Implementasi

  • Akar = CLI tunggal: sesuai arahan pembimbing, akar structure chart adalah python manage.py runserver sebagai satu kotak. Rantai Django boot sequence (manage.main → execute_from_command_line → Command → WSGIHandler → URL dispatcher) dijelaskan pada paragraf Pendahuluan dan tabel “Modul Boot & Dispatch”, tetapi tidak digambar di diagram karena merupakan infrastruktur framework, bukan modul aplikasi.
  • Tanpa autentikasi: seluruh endpoint publik read-only.
  • forecasts_api read-only: endpoint /api/forecasts/ hanya membaca dokumen forecasts/latest dari Firestore — tidak ada joblib.load, model XGBoost, atau scaler di jalur request.
  • Inferensi dipre-komputasi offline: trail forecast diproduksi oleh website/precompute_forecasts.py yang memanggil predictor.compute_forecast_trails dan menulis hasilnya ke D8 (forecasts/latest). prediction/feature_engineering.py tetap menjadi single source of truth fitur, dipakai bersama skrip training horizon_forecast_C{1..4}_*.py dan path precompute — agar tidak terjadi divergensi train/serve.
  • Composite-index avoidance: news meng-fetch semua doc dari news_articles lalu sort di Python, untuk menghindari kebutuhan composite index Firestore (date + sentiment).
  • Data store yang dibaca P5: D1 (daily_prices), D2 (news_articles), D3 (sentiment_aggregates), D4 (hmm_states), D7 (winners.json via web/winners.py), D8 (forecasts).
