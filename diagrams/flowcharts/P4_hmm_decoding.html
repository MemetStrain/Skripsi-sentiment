Flowchart — P4 HMM State Decoding

Alur kontrol detail untuk proses 4.0 — frozen-params HMM + online forward filter.

Pendahuluan

Proses P4 — HMM State Decoding diimplementasikan oleh scheduler/hmm_updater.py::update_hmm_states(db, write_existing=False). Fungsi ini tidak pernah me-refit HMM — parameter dimuat beku dari D5 (hmm_models/Daily) yang sebelumnya di-publish oleh skrip training offline (markov/cpo_hmm_states.py) via scheduler/migrate_hmm_to_firestore.py. Fitur direkonstruksi dari D1 (daily_prices), HMM di-instansiasi ulang dari params, dan online forward filter (bukan Viterbi) men-decode satu state per tanggal. Hasil ditulis ke D4 (hmm_states); tanggal yang sudah ada akan di-skip kecuali write_existing=True.

Flowchart ini menonjolkan tiga keputusan dengan early-return: (1) params tidak ditemukan (error path), (2) data harga terlalu sedikit (< 50 baris), dan (3) feature rows terlalu sedikit setelah preprocessing (< 30 baris). Plus filter per-baris yang menentukan dokumen mana yang benar-benar ditulis.

Catatan Implementasi

  • Forward filter (online), BUKAN Viterbi (offline): keputusan disengaja agar histori state tidak berubah seiring waktu. Viterbi smoothing akan memutakhirkan keseluruhan sequence setiap kali dijalankan (peek observasi di masa depan); forward filter hanya menggunakan O_1..O_t untuk state pada waktu t.
  • 5 fitur HMM (causal Z-score): Log_Return_Z, Volatility_Z, RSI_norm ((RSI−50)/50), MACD_Z, BB_Width_Z — semuanya direkayasa ulang dari close dengan window dari params['volatility_window'] dan params['norm_window'].
  • Mapping state→label: diambil dari params['state_to_label'] yang ditulis oleh training offline (bukan hardcoded di runtime). Mapping di sistem ini adalah {0: Bearish, 1: Bullish, 2: Neutral} atau urutan apa pun yang tersimpan di D5.
  • Guard data-cukup: < 50 baris harga dan < 30 baris fitur → early-return log.warning. Setelah preprocessing (rolling Z-score dengan min_periods = max(window // 2, 2)), beberapa baris warmup akan terbuang.
  • Idempotensi: default write_existing=False → setiap rerun hanya menambahkan dokumen untuk tanggal yang belum ditulis. Histori deterministik di bawah frozen-params, jadi re-write tidak diperlukan (kecuali params berubah → set write_existing=True secara eksplisit).
