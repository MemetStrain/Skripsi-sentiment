Decomposed Process View

Process 0 (the entire system) is decomposed into three internal processes: a read-only Web Frontend on Vercel, a Live Inference Engine embedded in the same Django app, and a Local Scheduler the maintainer runs ad-hoc on their workstation. The Frontend and Inference Engine read from Firestore and bundled model artefacts; only the Scheduler writes.

Notable Data Flows

A. Dashboard Page Load (1.0)
  1. Visitor → 1.0: GET /
  2. 1.0 reads D1 (90-day OHLCV), D2 (state labels filtered to frequency=Daily), D3 (sentiment aggregates), and D6 (winners.json for the metrics summary card and 4×7 metrics table).
  3. 1.0 renders dashboard.html with everything embedded as JSON in the page; the chart and trail overlay are drawn client-side.
  4. The browser then makes a follow-up GET /api/forecasts/?max_horizon=7&window_days=90 which routes into 2.0.
B. Forecast API Call (1.0 → 2.0)
  1. Visitor → 1.0: GET /api/forecasts/?max_horizon=7&window_days=90
  2. 1.0 invokes predictor.compute_forecast_trails(db, 7, 90) which is process 2.0.
  3. 2.0 reads D1, D2, D3, D6, D7 (see Level 2 for the breakdown), runs the chosen CSA model per horizon, and returns a JSON-serialisable dict.
  4. 1.0 forwards the dict as the HTTP response.
C. News Page Load (1.0)
  1. Visitor → 1.0: GET /news/?sentiment=Positive&page=2
  2. 1.0 streams all documents from D4 (no date filter), then filters and sorts in Python — this avoids needing a composite Firestore index.
  3. 1.0 paginates 9 cards per page, computes sentiment counts, and renders news.html.
D. Daily Scheduler Run (3.0)
  1. Maintainer → 3.0: python scheduler/main.py --mode daily
  2. 3.0 reads D8 to learn the latest local date.
  3. If stale, 3.0 fetches new data from Investing.com and MPOB, appends to D8, mirrors fresh rows into D1 (price) and D4 (articles).
  4. 3.0 recomputes aggregates from D8 + writes D3.
  5. 3.0 reads D5 (frozen HMM params), runs forward-filter decoding on the full price series, writes new state docs into D2.
  6. Predictions are not written. They are produced live by 2.0 on demand.
E. Initial Bulk Load (3.0)
  1. Maintainer runs --mode initial once to seed Firestore from D8.
  2. Each step (price → news → aggregates → HMM) is checkpointed in scheduler/initial_load_progress.json so the run is resumable.
