System Boundary Batas sistem yang dianalisis

👤 User (Pengguna Umum)

Deskripsi: Pengguna umum / investor yang ingin melihat prediksi harga CPO

Hak Akses:

  • Register akun baru
  • Login ke sistem
  • Melihat dashboard dengan grafik HMM
  • Melakukan prediksi harga
  • Melihat berita dengan sentimen analysis

Karakteristik: Read-only access, tidak bisa mengubah data sistem

👨‍💼 Admin

Deskripsi: Administrator sistem dengan hak akses penuh

Relationship: Inherits from User (Admin IS-A User)

Hak Akses:

  • Semua fitur User (Dashboard, Prediksi, Berita)
  • Upload data harga baru (CSV)
  • Trigger HMM state recalculation
  • Manage database & system settings

Karakteristik: Full CRUD access, dapat memodifikasi data sistem

⏰ Scheduler (System Timer)

Deskripsi: Background process otomatis (Cron Job / Task Scheduler)

Jenis: Secondary Actor (Non-human actor)

Fungsi:

  • Scraping berita dari MPOB secara otomatis (harian)
  • Menjalankan FinBERT sentiment analysis
  • Update News_Sentiment database
  • Tidak memerlukan login/autentikasi

Jadwal: Runs daily at 00:00 UTC

👥 Actor Hierarchy

Inheritance Relationship:
Admin -.-> User
Admin mewarisi semua capabilities dari User. Ini berarti Admin dapat mengakses semua use case yang dapat diakses oleh User, PLUS use case tambahan yang eksklusif untuk Admin.
Implication:
Saat mendesain sistem, setiap fitur yang dibuat untuk User otomatis tersedia untuk Admin. Admin tidak perlu koneksi eksplisit ke UC03, UC04, UC05 karena sudah inherited dari User.

UC04

🔮 Melakukan Prediksi

Tujuan: Memprediksi harga CPO untuk periode t+n hari ke depan
Input: Jumlah hari prediksi (t+n), pilihan model (RF/XGBoost)
Output: Prediksi harga, metrik (MAPE, R², MAE), visualisasi grafik
Include: UC01 (Login)
Process: Load pre-trained model → inference → visualize

User Admin (inherited)

Output: List berita, sentiment label (Positif/Negatif/Netral), sentiment score
Include: UC01 (Login)

Interaction: User dapat klik berita untuk melihat detail lengkap

User Admin (inherited)

Input: CSV file dengan kolom (Date, Open, High, Low, Close)
Output: Database updated, HMM states recalculated
Include: UC01 (Login) + role check (Admin only)

🔗 Association Relationships

User → UC02 (Register):
User dapat mendaftar akun baru. Ini adalah use case publik (tidak memerlukan login).
User → UC03 (Dashboard):
User dapat melihat dashboard. Use case ini include UC01 (Login), artinya user harus login dulu.
User → UC04 (Prediksi):
User dapat melakukan prediksi. Memerlukan login terlebih dahulu.
User → UC05 (Berita):
User dapat melihat berita dengan sentimen. Memerlukan login.
Admin → UC06 (Update Data):
Hanya Admin yang dapat mengupload data harga baru. Memerlukan login + role validation.
Scheduler → UC07 (Scraping Otomatis):
Background process yang berjalan otomatis tanpa interaksi user. Tidak memerlukan login.

⬅️ Generalization (Inheritance)

Admin -.-> User (inherits):
Admin adalah spesialisasi dari User. Admin mewarisi semua hak akses User (UC03, UC04, UC05) PLUS memiliki hak akses tambahan (UC06).
Notasi UML:
Panah dengan kepala segitiga kosong (hollow triangle) menunjuk dari child (Admin) ke parent (User).
Implementasi:
Dalam sistem, role Admin memiliki permission level lebih tinggi. Saat Admin login, sistem memberikan akses ke semua fitur User + fitur admin (Update Data).

📎 Include Relationships

UC03, UC04, UC05, UC06 ..> UC01 (include):
Semua use case utama (Dashboard, Prediksi, Berita, Update Data) WAJIB melakukan Login terlebih dahulu. Include adalah dependency yang mandatory.
Notasi:
Panah putus-putus dengan stereotype <<include>> menunjuk dari use case yang memerlukan ke use case yang diperlukan (UC01).
Implementasi:
Dalam kode, ini diimplementasikan sebagai middleware/guard yang mengecek autentikasi sebelum mengakses halaman/API tertentu.
