Inference Pipeline (decomposition of Process 2.0)

Triggered by GET /api/forecasts/?max_horizon=7&window_days=90. Five internal sub-processes turn raw Firestore data into a list of rolling forecast trails (one per horizon). All file-loading sub-processes cache via functools.lru_cache so repeat calls within the same Vercel function instance are essentially free.

Per-Request Step-by-Step

  1. Boundary check. 1.0 clamps max_horizon ∈ [1,7] and window_days ∈ [7,365], then calls 2.4.
  2. Winners lookup (2.1). First call: read winners.json from disk and cache it for the lifetime of the Vercel function instance. Subsequent calls return the cached dict.
  3. Feature frame (2.2). Pull all daily_prices docs and merge with hmm_states + sentiment_aggregates. Recompute price-derived (RSI, MACD, etc.) and HMM-derived features locally — this re-uses prediction/feature_engineering.py, the same code path that produced the offline training CSVs. Stored Firestore values for HMM-derived features are ignored on purpose to avoid scale mismatches.
  4. Per-horizon inference loop (2.4). For each h ∈ {1..max_horizon}:
    • Pick tag = winners[h] (e.g. "cpo_hmm"). Skip horizon if no winner.
    • Call 2.3 to fetch (model, scaler, feature_cols) from D7. Cache hit on warm processes.
    • Walk every row in the trailing window_days window:
      • Slice the row's feature columns into a 1×N matrix (select_feature_matrix). Skip if any NaN.
      • If a scaler is present: X = scaler.transform(X).
      • log_ret = model.predict(X)[0]; clip to ±10 for numerical safety.
      • pred_price = anchor_close × exp(log_ret).
      • Map the predicted date: if i + h ≤ last_idx, use the actual trading date; otherwise use the next weekday after the last actual (so all horizons converge at one shared future point).
  5. Format response (2.5). Bundle trails, winners metadata, and the full metrics matrix from winners.json into the response JSON.
  6. Return. 1.0 wraps the dict in JsonResponse. The browser plots each trail as a dashed line on the price chart and renders the horizon legend.

Design Decisions Visible in This DFD

  • Re-derive HMM-side features instead of trusting Firestore. 2.2 only consumes HMM_State + HMM_State_Label from D2 and recomputes the technical features locally. Firestore stores raw indicator values; the offline training CSV stored z-scored versions on a 252-day rolling window. Mixing the two breaks the model. Recomputing is cheap.
  • Module-level lru_cache on file/joblib loads. Vercel reuses warm function instances; without caching, every request would pay the full joblib.load hit on 7 horizons × 1 winning model = 7 .pkl files (~40 MB total). With caching, only the first request after a cold start is slow.
  • Live inference rather than precomputed predictions. The scheduler does not write a predictions collection. Doing so would mean the deployed app shows stale predictions until the next scheduler run, and the 4×7 metrics table couldn't reflect on-the-fly config changes. Live inference is acceptable because each horizon's call is <100 ms after warmup.
  • Trail truncation at the shared future point. Every horizon's last predicted date lands on the next trading day after the latest actual. Longer-horizon trails simply start later in the window so they all converge — this is what makes the chart's overlay visually coherent.
